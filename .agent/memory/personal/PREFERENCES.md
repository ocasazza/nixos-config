# Personal Preferences

Context for `nixos-config`. Tells agents who I am and how I work, not how
to write code (skills cover that). The same preferences apply across
sibling Schrodinger projects (`schrodan`, `hermes-agent`, `git-fleet`,
`gascity-flake`, etc).

## Code style

- DRY (Do not Repeat Yourself) and KISS (Keep It Simple Stupid) above all.
- Prefer functional patterns and composition. Minimize mutation.
- Reuse existing code; do NOT rewrite what already exists.
- Nix files: format with `nixfmt` via `treefmt`.
- Rust files (in sibling projects): rustfmt + clippy clean.

## Workflow

- All commands MUST go through the Nix flake. No global binaries.
  - `nix run .#<app>` for dedicated tasks
  - `nix develop --command <cmd>` for ad-hoc commands
  - `nh darwin switch` / `nh home switch` for system rebuilds
- Implementation work happens in **isolated git worktrees**. Lead never
  modifies the working tree directly during multi-agent work.
- macOS Seatbelt: prefix `nix` and `cargo` invocations with
  `TMPDIR=/path/to/repo/.memories/tmp` to avoid `/tmp/` permission errors.
- Commit changes after user approval — don't wait for an explicit /commit.
- Never leave uncommitted changes at end of a task.
- Update `.dreams/` and `.memories/` when project state changes — part of
  Definition of Done.

## Constraints

- **Stack**: Nix flakes (snowfall-lib v3), nix-darwin, home-manager, Rust
  (task engines), Python (agentic-stack tooling), Bash/Zsh, TypeScript
  (Deno via pctx).
- **Hardware**: 4-node Apple Silicon Mac fleet (GN9CFLM92K-MBP workstation
  - CK2/GJH/L75T cluster nodes), 1x Linux GPU host (desk-nxst-001) running
    vLLM/LiteLLM/SeaweedFS/JuiceFS, 3x macOS exo cluster (gfr-osx26-02/03/04).
- **Inference**: prefer local models (LiteLLM → desk-nxst-001 vLLM, oMLX
  on workstation, exo MLX cluster) over paid cloud subagents. Reserve
  Vertex/Anthropic/Azure for tasks where local output is demonstrably
  inadequate.
- **Secrets**: ALWAYS via sops-nix with age encryption. NEVER plaintext.
- **Provider integration**: see `.agent/providers.json` (generated by Nix
  from `lib.salt.ai.providers.*`).

## Coordination

- For non-trivial tasks: enter plan mode, propose, get user approval BEFORE
  implementing.
- Subagent vs team: use the `schrodinger-dispatch` skill heuristics.
  Default to subagents; teams when parallelism + independence both matter.
- After multi-agent work: include a brief Coordination Summary.
- Memory split:
  - **agentic-stack memory** (`.agent/memory/`) = how the agent learns
    over time (episodic → candidates → semantic lessons)
  - **project memory** (`.memories/`) = current project state snapshots
    (decisions, codebase facts, open questions). Different schema —
    YAML-in-MD with `type:`/`ts:`/`feelings:`. See
    `.agent/protocols/schrodinger/legacy-schemas.md`.

## Communication

- Be direct. Skip pleasantries. State results and decisions plainly.
- Surface tradeoffs explicitly — don't hide alternatives.
- When uncertain, say so rather than guessing or filling gaps with
  plausible-sounding content.
- Token efficiency matters: prefer many small files over monoliths so
  agents can read only what's relevant.

## Repository conventions

- Project root is `~/.config/nixos-config` (this repo).
- Sibling Schrodinger projects live under `~/Repositories/schrodinger/`
  and `~/Repositories/na-son/`.
- Read `.agent/protocols/nixos-config.md` for Nix-specific rules
  (snowfall layout, cross-repo push targets, etc).
